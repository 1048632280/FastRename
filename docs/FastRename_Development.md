# FastRename 项目开发文档

**版本：** v1.0  
**状态：** 待开发  
**目标平台：** Windows  
**技术栈：** Tauri + Rust + React + TypeScript

---

## 1. 项目目标

FastRename 是一款 Windows 桌面端本地文件重命名工具，面向需要高频、精准修改文件名局部内容的场景。首版目标是提供类似文本编辑器的文件名编辑体验：用户点击文件名中的任意字符位置即可编辑，保存后立即真实重命名文件，并通过撤销栈保障误操作可恢复。

首版不做通用文件管理器，不提供目录树、地址栏、递归扫描、批量确认应用等能力。产品核心是“单目录文件列表 + 精准编辑 + 即时落盘 + 可撤销”。

---

## 2. 首版范围

### 2.1 支持能力

- Windows 单平台桌面应用。
- 用户选择一个文件夹后，展示该目录下的文件。
- 支持拖拽文件夹进入应用并加载该目录。
- 支持拖拽单个文件进入应用，并加载该文件所在目录。
- 拖拽多个项目时，取第一个可解析目录；如果第一个是文件，则加载其所在目录。
- 只展示普通文件，默认不展示隐藏文件和系统文件。
- 设置项提供“显示隐藏/系统文件”开关。
- 不展示子目录，不递归扫描。
- 不支持文件夹重命名。
- 文件列表展示主文件名、扩展名、大小、修改时间。
- 列表支持按主文件名、扩展名、大小、修改时间排序。
- 表格列宽支持拖动调整。
- 主文件名可编辑，扩展名完全只读。
- 保存时立即调用系统文件重命名能力真实落盘。
- 支持应用运行期间的连续撤销。

### 2.2 不支持能力

- macOS / Linux。
- 目录树、地址栏、面包屑导航。
- 递归展示子目录文件。
- 文件夹展示与文件夹重命名。
- 扩展名编辑。
- 批量预览、批量应用、确认修改按钮。
- 关闭应用后的撤销历史持久化。

## 3. 技术方案

### 3.1 总体架构

应用采用 Tauri 桌面架构：

- 前端：React + TypeScript，负责文件列表、编辑交互、排序、列宽拖拽、提示状态与设置面板。
- 后端：Rust，负责目录读取、文件元数据解析、Windows 文件名规则校验、真实重命名、撤销重命名。
- 通信：前端通过 Tauri command 调用 Rust 能力。Rust 返回结构化结果，前端只负责状态渲染与交互编排。

### 3.2 推荐目录结构


## 4. 核心数据模型

### 4.1 文件条目

```ts
type FileEntry = {
  id: string;
  directory: string;
  stem: string;
  extension: string;
  fullName: string;
  sizeBytes: number;
  modifiedAt: string;
  isHidden: boolean;
  isSystem: boolean;
};
```

说明：

- `stem` 是不含扩展名的主文件名，是唯一可编辑字段。
- `extension` 不包含点号，例如 `pdf`、`docx`。
- 无扩展名文件的 `extension` 为空字符串。
- `id` 可使用完整路径或后端生成的稳定标识。重命名后必须同步更新。

### 4.2 重命名请求

```ts
type RenameRequest = {
  directory: string;
  oldFullName: string;
  desiredStem: string;
  extension: string;
};
```

### 4.3 重命名结果

```ts
type RenameResult = {
  oldFullName: string;
  newFullName: string;
  finalStem: string;
  extension: string;
  conflictResolved: boolean;
  warning?: string;
};
```

### 4.4 撤销记录

```rust
struct RenameHistoryItem {
    directory: PathBuf,
    from_name: String,
    to_name: String,
    timestamp: SystemTime,
}
```

含义：

- `from_name` 是重命名前名称。
- `to_name` 是重命名后名称。
- 执行撤销时尝试将 `to_name` 恢复为 `from_name`。
- 如果恢复名冲突，按同样的自动编号规则恢复为可用名称。

## 5. 交互设计

### 5.1 加载目录

用户可通过两种方式加载文件：

- 点击“选择文件夹”按钮，打开系统目录选择器。
- 将文件夹或文件拖入应用窗口。

加载规则：

- 仅读取目标目录的第一层普通文件。
- 默认过滤隐藏文件和系统文件。
- 开启“显示隐藏/系统文件”后，隐藏文件和系统文件参与展示。
- 子目录直接忽略。
- 读取后按当前排序规则展示，默认按主文件名升序。

### 5.2 文件列表

列表列定义：

| 列 | 是否可编辑 | 是否可排序 | 说明 |
| --- | --- | --- | --- |
| 主文件名 | 是 | 是 | 单击文字位置进入编辑 |
| 扩展名 | 否 | 是 | 完全只读，不显示点号 |
| 大小 | 否 | 是 | 以友好单位展示 |
| 修改时间 | 否 | 是 | 使用中文本地格式 |

列宽规则：

- 每列有默认宽度、最小宽度和最大宽度。
- 用户可拖动表头分隔线调整宽度。
- 主文件名列应优先获得剩余空间。
- 列宽调整不改变排序和编辑状态。

### 5.3 精准编辑

用户单击主文件名文字区域时，立即进入编辑状态：

- 光标落点应尽量匹配用户点击的字符间位置。
- 不允许默认全选整个主文件名。
- 只编辑主文件名，不编辑扩展名和点号。
- 当前行进入编辑态后，应保持扩展名列、大小列、修改时间列可见。

推荐实现：

- 使用 HTML input 或 contenteditable 承载编辑态。
- 进入编辑态时根据点击位置计算 caret offset。
- 如果浏览器能力导致像素级落点复杂，可优先采用 `contenteditable` + Selection API；若使用 input，则需要基于文本测量计算光标位置。

### 5.4 键盘流

| 按键 | 行为 |
| --- | --- |
| Enter | 保存当前编辑，进入下一行编辑态 |
| Tab | 保存当前编辑，进入下一行编辑态 |
| Shift+Tab | 保存当前编辑，进入上一行编辑态 |
| ↑ | 保存当前编辑，进入上一行编辑态 |
| ↓ | 保存当前编辑，进入下一行编辑态 |
| Esc | 放弃当前未保存输入，恢复原主文件名并退出编辑态 |
| Ctrl+Z | 非编辑态下撤销最近一次已落盘重命名 |
| Cmd+Z | 作为兼容预留，Windows 首版不作为主路径 |

编辑态内的普通文本撤销由输入控件自身处理。当前输入框没有可撤销文本操作时，`Ctrl+Z` 可触发全局重命名撤销。实现时需要避免和浏览器默认撤销行为冲突。

### 5.5 静默保存

触发保存的场景：

- 按 Enter。
- 按 Tab / Shift+Tab。
- 按 ↑ / ↓。
- 点击另一个文件。
- 点击列表空白区域。
- 当前编辑控件失去焦点。

保存规则：

- 内容未变化时不调用重命名。
- 内容为空时不落盘，恢复原名并给出短暂错误提示。
- 内容非法时不落盘，保留编辑态并提示。
- 内容合法但重名时，后端自动追加编号。
- 保存成功后刷新当前文件条目，并压入撤销历史栈。

## 6. 文件名校验

### 6.1 非法字符

Windows 文件名禁止包含：

```text
\ / : * ? " < > |
```

编辑时应即时拦截这些字符：

- 键盘输入非法字符时，不写入输入框。
- 输入框边框短暂红色闪烁。
- 可显示 1 秒轻量提示。

### 6.2 粘贴处理

粘贴文本时：

- 自动过滤非法字符。
- 过滤后为空时拒绝粘贴或拒绝保存，并提示。
- 粘贴结果仍需经过完整 Windows 文件名规则校验。

### 6.3 完整 Windows 规则

保存前必须校验：

- 主文件名不能为空。
- 主文件名不能只包含空白字符。
- 文件名末尾不能是空格或点号。
- 禁止 Windows 保留名：`CON`、`PRN`、`AUX`、`NUL`、`COM1` 到 `COM9`、`LPT1` 到 `LPT9`。
- 保留名大小写不敏感。
- 带扩展名时仍需识别保留名，例如 `CON.txt` 的主名 `CON` 不允许。

校验职责以 Rust 后端为准。前端可做即时提示，但不能替代后端最终校验。

## 7. 重名冲突处理

### 7.1 实时预警

用户编辑过程中，前端根据当前目录文件列表判断是否存在同名文件：

- 比较完整文件名，即 `主文件名 + 扩展名`。
- 排除当前正在编辑的原文件。
- Windows 下比较大小写不敏感。
- 发现冲突时输入框边框或文字显示橙色。

### 7.2 保存时自动编号

如果保存时发现目标名称已存在，后端自动生成可用名称：

```text
测试报告.txt
测试报告(1).txt
测试报告(2).txt
```

规则：

- 编号追加到主文件名末尾，扩展名保持不变。
- 从 `(1)` 开始递增。
- 如果用户输入本身已带编号，也仍按最终完整名称查找下一个可用名称。
- 自动编号成功后不弹窗，只给轻量提示。

## 8. 撤销机制

### 8.1 历史栈

Rust 后端在内存中维护重命名历史栈。每次真实重命名成功后压入记录。

不入栈的情况：

- 文件名未变化。
- 保存失败。
- 空文件名被拒绝。
- 非法文件名被拒绝。

### 8.2 执行撤销

用户按 `Ctrl+Z` 时：

1. 前端调用后端撤销命令。
2. 后端弹出最近一条历史记录。
3. 检查 `to_name` 当前是否仍存在。
4. 如果存在，尝试恢复为 `from_name`。
5. 如果 `from_name` 冲突，自动恢复为 `from_name` 的编号版本。
6. 返回撤销结果，前端刷新列表。

如果 `to_name` 已不存在，说明文件被外部移动、删除或再次修改。此时不弹窗阻断，只显示轻量错误提示，并继续允许用户后续操作。

## 9. Tauri Command 设计

### 9.1 目录读取

```rust
#[tauri::command]
fn list_files(directory: String, options: ListFilesOptions) -> Result<Vec<FileEntry>, AppError>
```

职责：

- 校验目录存在。
- 读取单层文件。
- 忽略目录。
- 根据设置过滤隐藏和系统文件。
- 返回文件名、扩展名、大小、修改时间等元数据。

### 9.2 重命名

```rust
#[tauri::command]
fn rename_file(request: RenameRequest) -> Result<RenameResult, AppError>
```

职责：

- 校验主文件名。
- 拼接目标完整文件名。
- 检查冲突并自动编号。
- 调用 `std::fs::rename` 或 Windows 平台封装完成重命名。
- 写入内存撤销历史。
- 返回最终名称。

### 9.3 撤销

```rust
#[tauri::command]
fn undo_last_rename() -> Result<UndoResult, AppError>
```

职责：

- 从历史栈取出最近记录。
- 尝试恢复原名称。
- 冲突时自动编号。
- 返回最终恢复名称或错误。

### 9.4 设置

```rust
#[tauri::command]
fn get_settings() -> Result<AppSettings, AppError>

#[tauri::command]
fn save_settings(settings: AppSettings) -> Result<(), AppError>
```

首版设置至少包含：

```ts
type AppSettings = {
  showHiddenAndSystemFiles: boolean;
};
```

设置可持久化到 Tauri app config 目录。撤销历史不持久化。

## 10. 前端状态设计

### 10.1 主要状态

```ts
type SortState = {
  key: "stem" | "extension" | "sizeBytes" | "modifiedAt";
  direction: "asc" | "desc";
};

type EditingState = {
  fileId: string;
  originalStem: string;
  draftStem: string;
  caretOffset: number;
};

type ColumnWidths = {
  stem: number;
  extension: number;
  size: number;
  modifiedAt: number;
};
```

### 10.2 状态原则

- 文件列表以服务端返回为准。
- 排序由前端完成，重命名成功后保持当前排序规则。
- 编辑态只允许同时存在一个。
- 触发新编辑前必须先处理旧编辑的保存或放弃。
- 设置变化后重新加载当前目录。

## 11. UI 设计要求

首版界面应是工具型应用，而不是营销页。第一屏就是可使用的文件列表。

建议布局：

- 顶部工具栏：选择文件夹按钮、当前目录摘要、设置入口、撤销按钮。
- 主区域：文件表格。
- 空状态：提示选择或拖拽文件夹。
- 底部状态栏：文件数量、当前排序、最近一次操作提示。

视觉原则：

- 中文界面。
- 信息密度适中，适合反复操作。
- 编辑态边框清晰。
- 非法输入使用红色短反馈。
- 重名预警使用橙色短反馈。
- 避免阻断式弹窗，除非系统目录选择器本身需要。

## 12. 错误处理

错误提示必须轻量，不打断连续编辑。

常见错误：

- 目录不存在或不可访问。
- 文件读取失败。
- 文件被外部删除。
- 文件被其他程序占用导致重命名失败。
- 文件名非法。
- 撤销目标不存在。

处理策略：

- 优先在状态栏或 toast 显示。
- 不使用“是否保存”确认弹窗。
- 重命名失败时保持原列表状态，并允许用户继续编辑。
- 必要时提供刷新按钮重新读取目录。

## 16. 关键风险

- 精准光标落点实现复杂，需在早期做原型验证。
- 静默保存要求所有异常都不能阻断用户心流，因此错误提示设计要克制且明确。
- 文件可能被外部程序占用，后端必须正确返回失败状态。
- Windows 文件名规则比非法字符更复杂，不能只依赖前端校验。
- 排序后连续编辑的“下一行”含义应以当前可见排序顺序为准。
- 撤销只能保证应用运行期间的历史，关闭应用后不再可恢复。
